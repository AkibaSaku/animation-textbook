Python · ASGI Server

ASGIの受付ホールuvicorn

FastAPI などのPythonアプリは、それ単体ではネットの向こうと話せません。uvicorn は外からのHTTP接続を受け、ASGIという共通の作法に変換してアプリへ渡す受付ホールです。肝は1人の受付（イベントループ）が、待ち時間の発生する用件を並行で捌くこと。DB待ちの客を待たせている間に、次の客をどんどん進められます。

01どこに立っているのか

ネットとアプリの間。uvicornが「生のHTTP」を「ASGIイベント」に翻訳して受け渡します。

NETブラウザ / クライアント生のHTTPリクエスト
uvicorn受付ホール（ASGIサーバー）HTTP ⇄ ASGI を翻訳
ASGI共通インターフェース（約束事）scope / receive / send
APPFastAPI / Starlette などのアプリあなたの async 関数

ASGIとは： サーバーとアプリの標準の握手のしかた。これに従う限り、アプリ側はサーバーが何でも動くし、asyncやWebSocketも扱える（古い同期専用のWSGIとの違い）。

021人の受付が並行で捌く（イベントループ）

「リクエスト到着」を押すと客が来ます。各客はDB待ち（I/O）が発生し、受付はその間に他の客を進めます。

リクエストを到着させてください。待ち時間中も受付は止まりません

ここが非同期の効き目： 受付（イベントループ）は1人でも、await でDB応答を待つ客を脇に置いて次を進められる。CPUを使わない待ち時間が多い処理（I/O）ほど、少ないスレッドで大量を捌けます。

03実運用のかたち

起動コマンド

uvicorn main:app --reload。mainモジュールのapp（ASGIアプリ）を指して立ち上げる。

複数ワーカー

受付ホールをCPUコア数だけ並べると並列性が上がる。本番では gunicorn に uvicorn worker を載せる構成も定番。

全体のつながり： 前タブの Pydantic が型を守り、FastAPI がルーティングと OpenAPI 生成を担い、その一番外で uvicorn が世界とアプリをつなぐ。この3つが現代Python Webの定番セットです。
