Realtime · Server push

一方通行の実況中継Server-Sent Events

「サーバーからの更新だけ受け取れれば十分」という場面は多い——株価ティッカー、進捗バー、通知、LLMの逐次出力など。SSE は1本のHTTPレスポンスを閉じずに開いたまま、サーバーがテキストのイベントを流し続ける仕組み。WebSocketと違い受信専用の一方通行ですが、その分普通のHTTPのままで手軽、しかも切れたら自動で再接続します。

01流れ続けるレスポンス

Content-Type: text/event-stream のまま、data: 行を繰り返し書き込むだけ。形式はとても素朴です。

# レスポンスが閉じずに届き続ける id: 1 data: {"price": 120} id: 2 data: {"price": 123} retry: 3000 # 切れたら3秒後に自動再接続

02受信して、切れても自動で戻る

購読を開始するとイベントが流れます。「回線を切る」を押すと、SSEらしく自動再接続して途中から再開します。

未購読GET /events (text/event-stream)

気が利く点： ブラウザの EventSource が再接続を勝手にやり、最後に受け取った id を Last-Event-ID ヘッダで送るので、サーバーは続きから再開できる。実装はサーバーもクライアントもごく短く済みます。

03WebSocketとの選び分け

SSEで十分

更新を「受け取るだけ」。通知・進捗・フィード・LLMのトークン逐次表示。HTTPのままで楽。

WebSocketが必要

クライアントからも頻繁に送る双方向。チャット・ゲーム・共同編集。バイナリも扱える。

ひとこと： SSEはテキスト専用・同一サーバーへの同時接続数に上限あり（HTTP/1.1時）。とはいえ「サーバー→ブラウザの実況」が要件なら、最小コストの選択肢です。
